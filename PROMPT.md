# Task: Build Sseclone — AI video clipping tool (masterprompt)

Build the full product defined in `SPEC.md` by working through the phase
prompts in `prompts/`, in order. This file is the orchestrating loop prompt
for ralph-orchestrator (run with:
`ralph run -P PROMPT.md --completion-promise "ALL_PHASES_COMPLETE"`);
each phase file contains that phase's requirements, gates, and its own status
checklist.

## How this loop works

- The product spec is in `SPEC.md`. Read it every iteration. It is the only
  source of truth for features and stack; there is no conversation history.
- Each phase lives in `prompts/phase-NN-*.md` and ends with its own
  `- [ ] PHASE_COMPLETE` checkbox inside that file.
- A phase is done only when every acceptance criterion in its file passes AND
  its `PHASE_COMPLETE` box is checked in that file.
- Checkboxes in these files are DISK STATE for your future iterations, not
  the stop signal. The loop stops only on the completion signal defined at
  the bottom of this file.

## Iteration Rules

- Start every iteration with: re-read this file and `SPEC.md`, then run
  `git log --oneline -15` and `git status` to see prior work.
- Find the first unchecked phase in the Status list below. Open its prompt
  file and work ONLY on that phase's checklist. Do not start a later phase
  early, even if it looks easy.
- Within a phase: finish any in-progress checklist item before starting a new
  one. If an item won't fit in one context window, split it into smaller
  items inside the phase file and end the iteration without implementing.
- Global gates that must pass before ANY commit, in every phase:
  - `npm run lint` exits 0
  - `npm run typecheck` exits 0
  - `npm test` exits 0
- Never weaken, skip, or delete an existing test to make a gate pass. Tests
  are the behavioral contract. If a test is genuinely wrong, fix it and say
  why in the commit message.
- Commit after each working increment with message `phase-NN: <what changed>`.
  Never end an iteration with uncommitted work.
- When a phase's criteria all pass, check `- [x] PHASE_COMPLETE` in the phase
  file, check the phase box below, and commit both in the same commit.
- In your final output each iteration, state: which phase you worked on,
  which files changed, which checklist items flipped, and the exact gate
  commands you ran with their results. Never produce the same status message
  twice — always include this iteration's specific diff summary.

## Constraints (global — phases may add more, never less)

- Stack is fixed by `SPEC.md` § Tech stack. Do not introduce Redis, Docker,
  Postgres, paid APIs, API keys, or GPU dependencies.
- No binary media files in git. Test fixtures are generated by
  `scripts/make-fixtures.sh`.
- All FFmpeg invocations via `execa` arg arrays in `src/lib/ffmpeg/` only.
- Long-running media work happens in the worker via the `jobs` table — never
  inside a Next.js request handler.
- Keep `README.md` runnable at all times: a new machine following it must be
  able to `npm install && npm run fixtures && npm test`.

## Completion signal

Only when EVERY phase box above is checked AND
`npm run lint && npm run typecheck && npm test && npm run build` all exit 0
in this same iteration: commit, then end your output with exactly this
single line:

ALL_PHASES_COMPLETE

Never output that string in any other situation — not in summaries, code,
or commit messages. (Phase files mention a LOOP_COMPLETE signal for
standalone runs; under this masterprompt that string is ignored — do not
rely on it.)
